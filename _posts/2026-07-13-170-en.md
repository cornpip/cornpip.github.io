## Introduction

While trying to add the attention face mesh model to the [mediapipe_face_mesh](https://pub.dev/packages/mediapipe_face_mesh) package, I ended up **rebuilding the TFLite runtime binary itself with custom ops baked in**. This post is a record of that.

`face_landmark_with_attention.tflite` refines the eyes, lips, and irises in a single inference pass and gives you **478 landmarks** directly. The existing path was a 2-pass structure: run the mesh model (468), then run the iris model again. With this model it is one pass. I assumed I could just swap the model file. Instead I got this:

```
Encountered unresolved custom op: Landmarks2TransformMatrix.
Node number 192 (Landmarks2TransformMatrix) failed to prepare.
```

The problem was not the model. It was the **runtime**.

## What a custom op is

A `.tflite` file is ultimately a **computation graph**. Input tensors pass through ops like `CONV_2D → RELU → FULLY_CONNECTED → …` and become output tensors. The interpreter walks the graph and, for each op, looks up the corresponding **kernel implementation** to execute.

- **builtin op**: an operation built into TFLite as standard (`CONV_2D`, `ADD`, `SOFTMAX`, …). Identified by an enum code.
- **custom op**: an operation that is not in the standard set. It is embedded in the model as a **string name**, and to run it you must **register a kernel for that name at runtime**.

`.tflite` is a FlatBuffer format, and each op in the graph points into an `operator_codes` table. A builtin op is identified by `builtin_code` (an enum number); a custom op is identified by `builtin_code = CUSTOM` plus a **`custom_code` string**. So custom ops are **matched by name**. When the interpreter walks the graph and hits a `custom_code` it does not know, it asks the resolver for a kernel with that name — and if there is none, you get the `unresolved custom op` error above.

The strings really are sitting right there in the model file:

```bash
strings face_landmark_with_attention.tflite | grep -i transform
# TransformTensorBilinear
# TransformLandmarks
# Landmarks2TransformMatrix
```

### The three custom ops this model uses

| Op name | What it does |
|---|---|
| `Landmarks2TransformMatrix` | Computes a **transformation matrix** from landmark coordinates — e.g. deriving the crop/rotation transform for the eye region from the coarse mesh points around the eye. |
| `TransformTensorBilinear` | **Bilinearly resamples a feature map** according to that matrix. This is what aligns and crops the eye/lip regions. |
| `TransformLandmarks` | Applies the matrix **to landmark coordinates**, mapping refined region coordinates back into the full-face coordinate system. |

Together these three form the "attention" mechanism: produce a coarse mesh → compute a per-region transform (`Landmarks2TransformMatrix`) → warp that region's features (`TransformTensorBilinear`) → run a refinement head → map the result back to the original coordinate system (`TransformLandmarks`). That is why the eyes, lips, and irises come out more precise.

Note this is different from TFLite's **Flex (TF Select) ops**. Flex pulls in ops from TensorFlow proper; these are **separate kernels MediaPipe implemented in its own source**.

### What a kernel looks like

In TFLite, a single op kernel is a **bundle of function pointers** called `TfLiteRegistration`.

```c
typedef struct {
  void* (*init)(TfLiteContext*, const char* buffer, size_t length);  // parse options
  void  (*free)(TfLiteContext*, void* buffer);
  TfLiteStatus (*prepare)(TfLiteContext*, TfLiteNode*);   // compute output shape, prepare memory
  TfLiteStatus (*invoke)(TfLiteContext*, TfLiteNode*);    // the actual computation
  const char* custom_name;                                // "TransformTensorBilinear"
  ...
} TfLiteRegistration;
```

A function like `Register...V2()` returns this struct. In other words, **the custom op's logic is the body of `prepare`/`invoke`**, and that code has to be compiled into the binary.

## Why the stock runtime cannot run it

This package uses the TFLite **C API**. When the C API creates an interpreter, it internally calls:

```cpp
// tensorflow/lite/create_op_resolver.h
std::unique_ptr<MutableOpResolver> CreateOpResolver();
```

The **OpResolver** it returns is the "op name/code → kernel" lookup table. The **stock** build — meaning the untouched, as-shipped build TFLite officially distributes, which I will keep using as the baseline against my custom rebuilt binary — links `create_op_resolver_with_builtin_ops`, which hands back a resolver containing **builtin ops only**.

So the distributed `libtensorflowlite_c.so` and `TensorFlowLiteC.framework` contain **neither the implementation nor the registration** of those three custom ops.

```bash
strings libtensorflowlite_c.so | grep -c TransformTensorBilinear   # → 0
```

And nobody publicly ships a C API binary with these custom ops in it. The MediaPipe Tasks library has them, but its API is completely different. The only way out was to build it myself.

## The fix - swap out the resolver

Fortunately, the C API decides its whole op list through a single function: `CreateOpResolver()`. So if I **replace that function with a version registering "builtin + custom ops" and rebuild just the C binary**, the model loads without changing a single line of app code.

MediaPipe already has the registration function. [`mediapipe/util/tflite/cpu_op_resolver.cc`](https://github.com/google-ai-edge/mediapipe/blob/master/mediapipe/util/tflite/cpu_op_resolver.cc) registers seven custom ops:

```cpp
namespace mediapipe {
void MediaPipe_RegisterTfLiteOpResolver(tflite::MutableOpResolver* resolver) {
  resolver->AddCustom("MaxPoolingWithArgmax2D",
                      tflite_operations::RegisterMaxPoolingWithArgmax2D());
  resolver->AddCustom("MaxUnpooling2D",
                      tflite_operations::RegisterMaxUnpooling2D());
  resolver->AddCustom("Convolution2DTransposeBias",
                      tflite_operations::RegisterConvolution2DTransposeBias());
  resolver->AddCustom("TransformTensorBilinear",
                      tflite_operations::RegisterTransformTensorBilinearV2(),
                      /*version=*/2);
  resolver->AddCustom("TransformLandmarks",
                      tflite_operations::RegisterTransformLandmarksV2(),
                      /*version=*/2);
  resolver->AddCustom("Landmarks2TransformMatrix",
                      tflite_operations::RegisterLandmarksToTransformMatrixV2(),
                      /*version=*/2);
  resolver->AddCustom("Resampler", tflite_operations::RegisterResampler(),
                      /*version=*/1);
}
}  // namespace mediapipe
```

So the code I had to write is all of this:

```cpp
// mediapipe/custom_capi/create_op_resolver.cc
#include "mediapipe/util/tflite/cpu_op_resolver.h"
#include "tensorflow/lite/kernels/register.h"
#include "tensorflow/lite/mutable_op_resolver.h"

namespace tflite {

std::unique_ptr<MutableOpResolver> CreateOpResolver() {
  auto resolver = std::make_unique<ops::builtin::BuiltinOpResolver>();  // all builtins
  mediapipe::MediaPipe_RegisterTfLiteOpResolver(resolver.get());        // + custom ops
  return resolver;
}

}  // namespace tflite
```

This works because TFLite deliberately leaves an **official extension point** open. `tensorflow/lite/c/BUILD` has a variant target, `c_api_without_op_resolver`, that intentionally leaves `CreateOpResolver()` undefined:

```python
tags = ["allow_undefined_symbols"],  # For tflite::CreateOpResolver().
```

Which means: "define it yourself and link it in." Link the source above and you are done.

### Registration and implementation are different layers

This was the most confusing part. Looking at that code, you wonder: "where is the op's actual computation? I never wrote it."

The key is that using an op requires **two separate layers**:

| Layer | What it does | Do I write it? | Needed in the binary? |
|---|---|---|---|
| **Registration** | Maps name → kernel (`AddCustom(...)`) | Yes | Yes |
| **Implementation** | The body of `prepare`/`invoke` (the actual math) | No (MediaPipe provides it) | Yes (mandatory) |

`AddCustom` only **connects** a name to a function pointer. The **code that pointer points at must be compiled and linked from somewhere**, or you get `undefined reference: RegisterTransformTensorBilinearV2` at link time.

Bazel pulls that implementation in automatically through the deps chain:

```
custom_create_op_resolver
  └─ //mediapipe/util/tflite:cpu_op_resolver              (the registration function)
       └─ //mediapipe/util/tflite/operations:transform_tensor_bilinear
                                                          (the TFLite kernel: prepare/invoke)
            └─ @org_tensorflow//tensorflow/lite/delegates/gpu/common/mediapipe:
               transform_tensor_bilinear                  (attribute structs, parsers, GPU kernels)
```

The moment I put a single `cpu_op_resolver` line in deps, every `.cc` in that chain gets compiled along with it. So this is not "copy MediaPipe sources into the TFLite folder and build." It is **MediaPipe (kernels) + TF (dependency targets) + the TF C API shell**, linked in place by the MediaPipe build.

Had this been a **genuinely new custom op** rather than an existing MediaPipe implementation, I would have had to write the `prepare`/`invoke` logic in C++ myself. Here MediaPipe had already implemented it, so registration was all that was needed.

## Why the MediaPipe workspace, not vanilla TF

There is a trap here. That last line of the chain — the path `tensorflow/lite/delegates/gpu/common/mediapipe/` — **does not exist in TF master.**

It turns out these are **files MediaPipe patches into TF at build time**. MediaPipe's `WORKSPACE` pins `@org_tensorflow` to a specific commit and applies four patches:

```python
http_archive(
    name = "org_tensorflow",
    urls = ["https://github.com/tensorflow/tensorflow/archive/fad6b3cf5a7d51a437bd01ee929853bc8554b618.tar.gz"],
    patches = [
        "@//third_party:org_tensorflow_c_api_experimental.diff",
        "@//third_party:org_tensorflow_custom_ops.diff",     # ← this one plants them
        "@//third_party:org_tensorflow_objc_build_fixes.diff",
        "@//third_party:org_tensorflow_verifier_int4.diff",
    ],
    ...
)
```

`org_tensorflow_custom_ops.diff` **creates** `gpu/common/mediapipe/{BUILD, custom_parsers.cc, landmarks_to_transform_matrix.cc, transform_landmarks.cc, transform_tensor_bilinear.cc, ...}` inside the TF tree. So if you clone vanilla TF and try to build, that path simply is not there and the build fails.

The conclusion: **the MediaPipe workspace is the only place where the patched TF, the op sources, and the resolver all exist together.** So you build inside the MediaPipe repo and reference the C API targets as `@org_tensorflow//...`.

The whole picture:

```
[source prep]
  clone MediaPipe ── WORKSPACE fetches @org_tensorflow (pinned commit) + applies patches
                       └ custom_ops.diff → creates gpu/common/mediapipe/*
  + mediapipe/custom_capi/{create_op_resolver.cc, BUILD}   ← what I add

[build (bazel, MediaPipe workspace)]
  custom_create_op_resolver ─┐
  c_api_without_op_resolver ─┼─► link ─► C API binary with custom ops
                             ┘
       Android: cc_binary(linkshared) → libtensorflowlite_c.so (per ABI)
       iOS:     ios_static_framework  → TensorFlowLiteC.framework

[post-processing]
  strip / hide symbols → verify → swap in for the package's prebuilt binary

[runtime]
  load model → TfLiteInterpreterCreate → CreateOpResolver() (builtin + custom)
            → AllocateTensors() succeeds
```

## The Android build

TF ships `tensorflow/lite/tools/tflite-android.Dockerfile`, which sets up bazel + the Android SDK + NDK. The image provides only the *environment*; the sources and custom ops are mine to supply.

```bash
docker build . -t tflite-builder -f tflite-android.Dockerfile
docker run -d --name tflite-c \
  -v ~/attn-build/mediapipe:/mediapipe -v ~/attn-build:/host_dir \
  tflite-builder sleep infinity

docker exec tflite-c bash -lc '
cd /mediapipe
export ANDROID_HOME=/android/sdk ANDROID_NDK_HOME=/android/ndk
bazel build -c opt --config=android_arm64 --cxxopt=--std=c++17 \
    //mediapipe/custom_capi:libtensorflowlite_c.so
'
```

The traps I hit:

**1. The NDK toolchain is not registered in WORKSPACE.** MediaPipe's WORKSPACE only *loads* `android_ndk_repository` and never calls it (`# @unused`). Leave it as is and there is no Android toolchain, so the build fails. You have to add it:

```python
android_ndk_repository(name = "androidndk", api_level = 30)
register_toolchains("@androidndk//:all")
```

**2. The `select()` trap in the `tflite_cc_shared_object` macro.** At first I tried to use TF's macro exactly like stock does, but the `//tensorflow:...` labels inside the macro get misresolved in the MediaPipe repo context as `@//tensorflow` (i.e. `/mediapipe/tensorflow`) and break with "no such package". I dropped the macro and worked around it with a plain `cc_binary(linkshared = 1)`.

```python
cc_binary(
    name = "libtensorflowlite_c.so",
    linkshared = 1,
    linkopts = [
        "-Wl,--version-script,$(location @org_tensorflow//tensorflow/lite/c:version_script.lds)",
        "-Wl,-z,max-page-size=16384",
    ],
    deps = [
        ":custom_create_op_resolver",
        "@org_tensorflow//tensorflow/lite/c:c_api_without_op_resolver",
        "@org_tensorflow//tensorflow/lite/c:c_api_experimental_without_op_resolver",
        "@org_tensorflow//tensorflow/lite/c:version_script.lds",
    ],
)
```

**3. XNNPACK's newest ISA kernels fail on x86_64.** The clang in NDK r25b does not know flags like `-mavx512fp16` and `-mavxvnniint8`. Turning off just those kernels is enough:

```bash
--define=xnn_enable_avx512fp16=false --define=xnn_enable_avxvnniint8=false ...
```

**4. The output is 2.5x the size of stock.** My first build came out at 10MB, against stock's 4MB. "Did the custom ops bloat it?" No — **stock was simply already stripped.**

```bash
llvm-strip --strip-all libtensorflowlite_c.so
# arm64-v8a: 10MB → 4.1MB, x86_64: 11.5MB → 6.1MB (same as stock)
```

**5. 16 KB page alignment (the last trap to blow up).** I thought I was done, and then Android 15 warned that *"this app isn't 16 KB compatible."* The official TFLite release `.so` is linked with `-Wl,-z,max-page-size=16384`, so its LOAD segments are 16 KB (0x4000) aligned. My bazel build lacked that linkopt, so they came out 4 KB (0x1000) aligned.

[As of November 1, 2025](https://android-developers.googleblog.com/2025/05/prepare-play-apps-for-devices-with-16kb-page-size.html), Google Play requires new apps and updates targeting Android 15+ to support 16 KB page sizes on 64-bit devices. If you are in the business of replacing a prebuilt native library, this is not something you can wave off.

```bash
readelf -lW libtensorflowlite_c.so | grep LOAD   # Align must be 0x4000
```

**The lesson: when you swap out a prebuilt binary, you have to match stock not just in "contents" but in "link conditions."** iOS (Mach-O) has no such concept, so it does not apply there.

Verification looks at three things:

```bash
readelf -lW $f | grep LOAD                              # 16 KB alignment
strings -n6 $f | grep -c TransformTensorBilinear        # custom ops present
nm -D --defined-only $f | grep -c TfLiteXNNPackDelegate  # XNNPACK symbols retained
```

### Aside: does it actually run?

A name showing up in `strings` does not mean the op works. An arm64 `.so` cannot run on x86 Linux, so I also built the same custom C API for the host and ran a small C program that does `TfLiteInterpreterCreate → AllocateTensors` on the model. It got past exactly the point where stock fails, and produced **7 output tensors**:

| Output tensor | Size | Meaning |
|---|---|---|
| mesh | 1404 | 468 landmarks × 3 (x, y, z) |
| lips | 160 | 80 × 2 |
| left eye / right eye | 142 each | 71 × 2 |
| left iris / right iris | 10 each | 5 × 2 |
| face flag | 1 | face presence probability |

That confirms not just that the op is "in there" but that it "runs."

## The iOS build - this was the real mountain

**The idea is 100% the same as Android.** The resolver source is platform independent, so it is reused as is. The only difference is that the artifact is a `TensorFlowLiteC.framework` instead of a `.so`, and you need macOS + Xcode. Or so I thought. In practice it was far rougher.

### Plan: mirror the stock framework target → failed

Just like on Android, I tried to mirror the stock `//tensorflow/lite/ios:TensorFlowLiteC_framework` (the `tflite_ios_framework` macro) and swap only the resolver in its deps. Two walls stopped me.

**1. `BUILD.apple` is not a bazel package.** In TF, `tensorflow/lite/ios/` has a build file named **`BUILD.apple`**, not `BUILD`. Bazel therefore does not recognize the directory as a package. `load(".../tensorflow/lite/ios:ios.bzl")`, the header targets, `allowlist_TensorFlowLiteC.txt` — all of them fail with **"not a package"**.

**2. Visibility plus a dependency cycle.** That package's header targets and tools (`extract_object_files`, `hide_symbols_with_allowlist`) are visible only to `//tensorflow/lite:__subpackages__`, so `@mediapipe` cannot use them. Nor can I move my target into `@org_tensorflow`: my resolver depends on `//mediapipe/util/tflite:cpu_op_resolver`, which would create a **reverse TF → MediaPipe cycle**. That cycle also invalidates the "just patch TF's BUILD" alternative.

### Solution: assemble the framework by hand

Since TF's iOS macro was off limits, I assembled the framework **self-contained inside `@mediapipe`** using rules_apple's public `ios_static_framework`, referencing TF's ios package not at all. I copied the existing framework's `Headers/` in just to make the rule build, and in the final artifact I used **only the binary**, reusing the existing Headers/Modules.

```bash
JAVA_HOME=/opt/homebrew/opt/openjdk USE_BAZEL_VERSION=7.4.1 bazel build \
    --config=ios --ios_multi_cpus=arm64,x86_64 \
    -c opt --cxxopt=--std=c++17 \
    --repo_env=HERMETIC_PYTHON_VERSION=3.12 \
    --repo_env=JAVA_HOME=/opt/homebrew/opt/openjdk \
    //mediapipe/custom_capi:TensorFlowLiteC_framework
```

Every flag has a reason, and all of them were earned the hard way.

- **`--config=ios_fat` (what the official docs say) breaks.** `ios_fat` includes armv7/i386, but the recent rules_apple pulled in via bzlmod has no `ios_armv7` mapping, so you get a transition error (`key "ios_armv7" not found`). I replaced it with `--config=ios --ios_multi_cpus=arm64,x86_64`.
- **`HERMETIC_PYTHON_VERSION=3.12`** — the system python 3.14 is outside TF's `requirements_lock` range (3.9–3.12) and fails.
- **`JAVA_HOME` + `--repo_env=JAVA_HOME`** — without a JDK, rules_java's `local_jdk` repo breaks (`no such package @rules_java~//tools/jdk`) and the apple bundling analysis phase dies.

### 367MB → 18MB: hiding symbols was mandatory

The binary `ios_static_framework` produced was **367MB**. Stock is 47MB. The reason: it is just an **ar archive with every symbol global**. The stock `tflite_ios_framework` macro internally runs a **symbol merge/hide step**, and since I could not use that macro, the entire step was missing.

So I ran stock's script (`hide_symbols_with_allowlist.sh`) **manually, as a separate step**. What it does:

- Per architecture, run `ld -r -exported_symbols_list <allowlist> -x` to **merge every object into a single Mach-O**, and
- localize/strip every symbol outside the allowlist (`_TfLite*`, `*AcquireFlexDelegate*`).

Result: **18MB**, with exactly 219 exported `_TfLite*` symbols and zero non-TfLite externals. So it was never "adding custom ops made it bigger" — it was **the post-processing the macro used to do for me being absent**. It is the iOS counterpart of what a single `strip` handled on Android, only far more hands-on.

#### Why did it end up smaller than stock (47MB)?

The stock framework it replaced is 47.5MB; mine is 18.1MB. Both are arm64 + x86_64, so it is not a difference in the number of slices. Looking into it, two things compound:

1. **My link configuration is narrower than stock's.** My deps are just `c_api_without_op_resolver` + `c_api_experimental_without_op_resolver` + the resolver, while stock's `:tensorflow_lite_c` piles profiling/telemetry and more on top.
2. **Symbols and metadata shrank.** Stock carries the `SUBSECTIONS_VIA_SYMBOLS` flag and has ~2,520 symbol strings; mine lacks that flag and has ~646. Merging every object into a single Mach-O with `ld -r … -x` and stripping local symbols collapses the symbol and relocation tables. Less metadata, not less code.

Which means **"the framework file got smaller" does not equal "the app got smaller."** Without `SUBSECTIONS_VIA_SYMBOLS`, the final app linker may have a harder time dead-stripping at function granularity. If you are sensitive to shipped app size, compare final app binary size, not framework file size. This time I only verified linking and behavior; I did not measure the app-size impact.

### A trap in verification - "there are no symbols in the Runner binary"

I dropped the framework into `ios/Frameworks/`, built the example, checked the symbols — and saw **zero TfLite symbols in the Runner binary**. I assumed it had failed.

The cause was this package's loading model. On Apple, `src/tflite_runtime.h` resolves symbols with `dlsym(RTLD_DEFAULT, ...)` — a **static-linking model** — so the TfLite symbols land not in Runner but in **the plugin's dynamic framework** (`Runner.app/Frameworks/mediapipe_face_mesh.framework`). Looking there, `TfLiteInterpreterCreate` and the custom ops were present and correct.

### A remaining constraint: the simulator

A fat Mach-O can hold only one arm64 slice, so **arm64 = device** and **simulator = x86_64 (Rosetta)**. Stock has the same constraint. To run natively on an Apple Silicon simulator you would have to move to an **xcframework** with an arm64-sim slice — a separate task that changes the podspec's `vendored_frameworks` to a `.xcframework`.

> **Android vs iOS in one line**: on Android, dodging the macro's `select()` trap with a plain `cc_binary` was the whole job; on iOS, TF's iOS package was **entirely unreachable (BUILD.apple + visibility/cycle), so I had to assemble the framework myself and restore by hand the symbol hiding the macro used to do.**
{: .prompt-info }

## And the app side?

Once the runtime binary knows about the custom ops, the integration is ordinary.

- Add one option (`enableAttentionMesh`, off by default, so existing behavior is unchanged).
- When attention is on, **assemble the model's 7 output tensors into a 478 layout**: lay down the 468 mesh points, overwrite the eye and lip indices, and append the 10 iris points. The assembly rules follow MediaPipe's `tensors_to_face_landmarks_with_attention.pbtxt` configuration.
- Everything downstream — coordinate transforms, ROI, blendshapes — **cannot tell the difference; it just receives 478.**

So only the landmark-producing step was swapped; the rest was reused.

## Summary

- A `.tflite` is an op graph, and a **custom op** is an operation outside the standard set whose kernel you must register at runtime.
- The attention face mesh is implemented with **three MediaPipe custom ops**: transform-matrix computation, feature warping, and landmark back-transformation.
- The C API decides its op list via `CreateOpResolver()`, and stock has builtins only. → **Replace just that function and rebuild the C binary**, and the model loads with zero app-code changes. This is the official extension point TFLite exposes via `c_api_without_op_resolver`.
- That rebuild has to happen **inside the MediaPipe workspace (with its patched, pinned TF)**, because the custom ops' dependency files do not exist in TF master and only come into being through MediaPipe's patch.
- The custom ops themselves are plain TFLite kernels, so porting them was clean. What ate the time was **the build system, not the ops** — the macro `select()` trap and 16 KB alignment on Android; unreachable TF iOS packages and hand-restored symbol hiding on iOS.
- Size is easy to misread. Custom ops do not make it bigger; it only *looked* bigger because **I skipped the post-processing (strip / symbol hiding) that stock does for you**. Restore it and you get 4.1MB on Android (= stock) and 18MB on iOS (< stock's 47MB).
